The Four Fundamental Rules
Kubernetes mandates these networking requirements — CNI plugins must satisfy all four
01
Pod-to-Pod (same node)
Every pod can communicate with every other pod on the same node without NAT via a shared Linux bridge or veth pair.
02
Pod-to-Pod (across nodes)
Pods on different nodes can communicate directly using their pod IP — no NAT required. Handled by overlay or underlay routing.
03
Node-to-Pod
Nodes can reach any pod IP directly. Used by kubelet health checks, kube-proxy, and hostNetwork pods.
04
Unique Pod IPs
Every pod gets its own unique IP address from the cluster CIDR — no port collision between pods running the same image.

Service Types
Services provide stable virtual IPs and DNS names that load-balance across pod replicas
Default
ClusterIP
Creates a virtual IP reachable only inside the cluster. kube-proxy programs iptables/IPVS rules to load-balance to pod endpoints.
Use for
Internal service-to-service communication. Frontend → Backend API calls.
Client Pod
→ ClusterIP 10.96.0.1:80
→ iptables DNAT
→ Pod 10.244.x.x:8080
External
NodePort
Opens a static port (30000–32767) on every node's IP. External traffic hitting any node on that port is forwarded to the service.
Use for
Dev/test external access. Simple external exposure without a cloud LB.
Internet
→ NodeIP:30080
→ ClusterIP:80
→ Pod:8080
Cloud
LoadBalancer
Provisions an external cloud load balancer (AWS ALB, GCP CLB, Azure LB) with a public IP that routes to NodePort, then to pods.
Use for
Production external access on managed Kubernetes (EKS, GKE, AKS).
Internet
→ Cloud LB (203.x.x.1:80)
→ NodePort:30080
→ Pod:8080
Special
Headless
No cluster IP assigned (clusterIP: None). DNS returns individual pod IPs directly, enabling direct pod addressing for stateful apps.
Use for
StatefulSets (Kafka, Cassandra, etcd). Client-side load balancing.

CoreDNS — Cluster DNS

CoreDNS runs as a Deployment and provides DNS resolution for all Services and Pods within the cluster. Every pod's /etc/resolv.conf points to it.

DNS Records for Services
A my-svc.my-ns.svc.cluster.local → 10.96.0.1
SRV _http._tcp.my-svc.my-ns → port 80
PTR 1.0.96.10.in-addr.arpa → my-svc.my-ns
DNS Records for Pods
A 10-244-1-5.my-ns.pod.cluster.local → 10.244.1.5

DNS Resolution Path

How a pod resolves mysql.production.svc.cluster.local

App makes DNS query Resolver reads /etc/resolv.conf · search domain: cluster.local
Query sent to CoreDNS ClusterIP kube-dns Service IP: 10.96.0.10:53 (UDP)
CoreDNS looks up Service Reads from API server cache — finds mysql in production namespace
Returns ClusterIP A record: 10.96.14.23 — app connects to stable virtual IP
kube-proxy routes to pod iptables/IPVS DNAT rewrites destination to a healthy pod IP

kube-proxy Modes
kube-proxy watches Services and Endpoints; programs rules on each node to forward Service VIP traffic to healthy pods
Legacy

userspace

Traffic goes through a Go proxy process in userspace — slow due to kernel/userspace context switches. Deprecated.

High latency Bottleneck at proxy process Not used in modern clusters
Default

iptables

Programs iptables DNAT rules in the kernel. No userspace hops — random pod selection per connection. Default in most distros.

Kernel-speed forwarding No proxy process overhead
O(n) rule lookup at scale Random LB only (no least-conn)
Recommended

IPVS

Uses Linux IPVS (IP Virtual Server) with a hash table for O(1) rule lookup. Supports multiple load-balancing algorithms.

O(1) lookup regardless of scale Round-robin, least-conn, SH, DH Better throughput at 1000+ Services
